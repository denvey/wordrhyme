核心表结构

  user 表（所有用户）
  ├─ id
  ├─ email
  ├─ name
  ├─ role (全局角色: "admin", "auditor", null 等)
  └─ ...

  member 表（用户-组织关系）
  ├─ userId (关联 user.id)
  ├─ organizationId (关联 organization.id)
  ├─ role (组织角色: "owner", "admin", "member", "viewer")
  └─ ...

  organization 表（所有组织）
  ├─ id
  ├─ name
  └─ ...

  用户类型区分

  所有用户都在 同一张 user 表，通过以下方式区分：

  1. Platform 管理员（全局管理员）

  -- user 表
  user.role = "admin"

  -- member 表（Platform 组织成员）
  member.userId = xxx
  member.organizationId = "platform"
  member.role = "owner" 或 "admin"

  特点：
  - 拥有全局角色 user.role = "admin"
  - 是 Platform 组织的成员
  - 切换到 Platform 组织后，拥有跨租户权限

  2. 租户管理员（组织管理员）

  -- user 表
  user.role = null  (或其他非 admin 值)

  -- member 表（某个组织的管理员）
  member.userId = xxx
  member.organizationId = "test-org-id"
  member.role = "owner" 或 "admin"

  特点：
  - 没有全局角色（或有其他全局角色）
  - 是某个组织的 owner 或 admin
  - 只能管理自己组织的数据

  3. 普通用户

  -- user 表
  user.role = null

  -- member 表（某个组织的普通成员）
  member.userId = xxx
  member.organizationId = "test-org-id"
  member.role = "member" 或 "viewer"

  特点：
  - 没有全局角色
  - 是某个组织的普通成员
  - 只能访问自己组织的数据，权限受限

  用户可以属于多个组织

  一个用户可以在多个组织中拥有不同的角色：

  -- 用户 A 的成员关系
  member 表:
  | userId | organizationId | role   |
  |--------|----------------|--------|
  | user-a | org-1          | owner  |  -- 在 org-1 是管理员
  | user-a | org-2          | member |  -- 在 org-2 是普通成员
  | user-a | platform       | admin  |  -- 在 Platform 是管理员

  权限判断流程

  // 1. 获取用户的全局角色
  const globalRole = user.role;  // 如 "admin"

  // 2. 获取用户在当前组织的角色
  const orgRole = member.role;   // 如 "owner"

  // 3. 合并角色
  const userRoles = [globalRole, orgRole];  // ["admin", "owner"]

  // 4. 查询权限规则
  // - 查询当前组织的 "admin" 角色权限
  // - 查询当前组织的 "owner" 角色权限
  // - 合并所有权限

  // 5. 创建 CASL ability 实例
  const ability = createAppAbility(user, userRoles);

  // 6. 检查权限
  ability.can('read', 'User');  // true/false

  设计优点

  ✅ 简单统一：
  - 所有用户在同一张表，便于管理
  - 不需要维护多张用户表

  ✅ 灵活的角色系统：
  - 全局角色（user.role）：跨组织的身份
  - 组织角色（member.role）：组织内的身份
  - 两者可以独立配置

  ✅ 多租户支持：
  - 用户可以属于多个组织
  - 在不同组织中拥有不同权限

  ✅ 权限隔离：
  - 普通用户只能看到自己组织的数据
  - Platform 管理员切换到 Platform 组织后才能跨租户

  是否需要分表？

  不需要。当前设计已经足够：

  ❌ 不推荐分表的原因：
  - 增加复杂度（需要维护多张表）
  - 用户可能同时是管理员和普通用户（在不同组织）
  - 登录、认证逻辑会变复杂
  - 数据迁移困难（用户角色变化时）

  ✅ 当前设计的优势：
  - 单一用户表，简化认证
  - 通过角色字段区分用户类型
  - 灵活的权限组合
  - 符合 Better Auth 的设计理念

  示例场景

  场景 1：超级管理员

  用户 Alice:
  - user.role = "admin"
  - member 关系:
    - Platform 组织: role = "owner"
    - Test Org: role = "member"

  权限:
  - 在 Platform 组织: 拥有跨租户权限，可以查看所有组织数据
  - 在 Test Org: 只是普通成员，只能查看 Test Org 数据

  场景 2：多组织管理员

  用户 Bob:
  - user.role = null
  - member 关系:
    - Org A: role = "owner"
    - Org B: role = "admin"
    - Org C: role = "member"

  权限:
  - 在 Org A: 拥有所有权限（owner）
  - 在 Org B: 拥有管理权限（admin）
  - 在 Org C: 只有普通成员权限

  场景 3：普通用户

  用户 Charlie:
  - user.role = null
  - member 关系:
    - Org A: role = "member"

  权限:
  - 只能访问 Org A 的数据
  - 权限受限于 member 角色
