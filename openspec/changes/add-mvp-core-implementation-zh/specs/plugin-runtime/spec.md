# 插件运行时规范 (Plugin Runtime Specification)

## 新增要求

### 要求：插件清单验证 (Plugin Manifest Validation)

系统应扫描 `/plugins` 目录以查找 `manifest.json` 文件。每个清单必须符合以下模式：`{ pluginId, version, vendor, type, runtime, engines.nebula, capabilities, permissions?, server, admin }`。无效的清单应被拒绝并返回描述性错误。验证应使用 Zod 模式执行。

#### 场景：接受有效的清单
- **当** 插件目录包含有效的 `manifest.json` 时
- **那么** 插件在 `plugins` 表中注册
- **并且** 插件状态设置为 `enabled`
- **并且** 插件元数据对系统可用

#### 场景：拒绝无效的清单
- **当** 插件清单缺少 `pluginId` 字段时
- **那么** 插件被标记为 `invalid`
- **并且** 创建一个包含验证错误的审计日志条目
- **并且** 插件不被加载

#### 场景：Zod 验证捕获类型错误
- **当** 插件清单的 `version: 123` (数字而非字符串) 时
- **那么** Zod 验证失败并返回描述性错误
- **并且** 插件被标记为 `invalid`

#### 场景：版本不匹配被拒绝
- **当** 插件声明 `engines.nebula: "0.2.x"` 但核心版本为 `0.1.x` 时
- **那么** 插件在依赖解析期间被禁用
- **并且** 记录指示版本不兼容的错误

---

### 要求：插件生命周期管理 (Plugin Lifecycle Management)

插件应支持生命周期钩子：`onInstall`, `onEnable`, `onDisable`, `onUninstall`。系统必须在适当的时间调用这些钩子。生命周期钩子必须在错误边界内执行 (插件错误不应导致系统崩溃)。

#### 场景：安装生命周期 (Install lifecycle)
- **当** 首次安装插件时
- **那么** 调用 `onInstall` 钩子 (如果已定义)
- **并且** 调用 `onEnable` 钩子
- **并且** 插件在数据库中标记为 `enabled`

#### 场景：启用生命周期 (Enable lifecycle)
- **当** 禁用的插件被启用时
- **那么** 调用 `onEnable` 钩子
- **并且** 插件状态更新为 `enabled`

#### 场景：禁用生命周期 (Disable lifecycle)
- **当** 启用的插件被禁用时
- **那么** 调用 `onDisable` 钩子
- **并且** 插件状态更新为 `disabled`
- **并且** 插件的路由和 UI 被注销

#### 场景：卸载生命周期 (Uninstall lifecycle)
- **当** 卸载插件时
- **那么** 调用 `onDisable` 钩子 (如果已启用)
- **并且** 调用 `onUninstall` 钩子
- **并且** 插件文件从 `/plugins` 中移除
- **并且** 插件从数据库中移除

#### 场景：生命周期钩子错误隔离
- **当** 插件的 `onEnable` 钩子抛出错误时
- **那么** 捕获并记录该错误
- **并且** 插件被标记为 `crashed`
- **并且** 其他插件继续正常加载
- **并且** 系统不会崩溃

---

### 要求：受控运行时执行与配额 (Controlled Runtime Execution & Quotas)

所有插件执行（生命周期钩子与请求处理器）必须在 Runtime Adapter 之后运行，并实现 `RUNTIME_GOVERNANCE.md` 中冻结的 `PluginRuntime` 抽象接口。适用于 MVP 的最小实现必须至少强制：执行超时、每插件最大并发任务数。超过限制必须终止任务，并且可以隔离/禁用插件。

#### 场景：生命周期钩子超时
- **当** 插件的 `onEnable` 在配置的超时内未完成
- **那么** 运行时终止该任务
- **并且** 插件被标记为 `crashed` 或 `disabled`
- **并且** 系统继续启动

#### 场景：并发限制生效
- **当** 插件已达到最大并发任务数
- **并且** 又调度了一个该插件的新任务
- **那么** 运行时拒绝或排队该任务（实现可定义）
- **并且** 系统保持稳定

---

### 要求：插件隔离 (Plugin Isolation)

插件必须彼此隔离，并与核心内部逻辑隔离。插件不得访问其他插件的状态或数据。插件应仅通过能力 API (`@nebula/plugin-api`) 与核心交互。

#### 场景：插件无法访问核心内部
- **当** 插件尝试 `import '@nebula/core/internal'` 时
- **那么** 导入失败 (找不到模块或拒绝访问)
- **并且** 插件无法加载

#### 场景：插件无法访问另一个插件的数据
- **当** 插件 A 尝试通过直接访问数据库读取插件 B 的数据时
- **那么** 查询返回空结果 (执行租户 + 插件隔离)
- **或者** 能力 API 阻止查询 (能力未声明)

---

### 要求：插件能力白名单 (Plugin Capability White-listing)

插件应仅访问其清单中 `capabilities` 部分声明的能力。未声明的能力必须在注入时被拒绝。

#### 场景：授予声明的能力
- **当** 插件在其清单中声明了 `capabilities.data.read: true` 时
- **那么** 插件在 `onEnable` 期间收到一个 Data 能力实例
- **并且** 插件可以调用 `ctx.data.read(...)`

#### 场景：拒绝未声明的能力
- **当** 插件未在其清单中声明 Data 能力时
- **那么** 插件不会收到 Data 能力实例
- **并且** 尝试调用 `ctx.data.*` 将抛出错误

---

### 要求：插件依赖解析 (Plugin Dependency Resolution)

系统应基于插件清单构建依赖图。循环依赖应被拒绝。冲突的插件不应同时启用。

#### 场景：解析有效的依赖关系
- **当** 插件 A 声明 `dependencies: ["plugin-b@1.x"]` 时
- **并且** 安装了插件 B 1.2.0 版本
- **那么** 两个插件均成功加载
- **并且** 插件 B 在插件 A 之前加载

#### 场景：拒绝循环依赖
- **当** 插件 A 依赖插件 B 时
- **并且** 插件 B 依赖插件 A 时
- **那么** 两个插件均被标记为 `invalid`
- **并且** 记录指示循环依赖的错误

#### 场景：冲突插件被禁用
- **当** 插件 A 声明 `conflicts: ["plugin-b"]` 时
- **并且** 同时安装了插件 A 和插件 B 时
- **那么** 系统禁用其中之一 (基于优先级或安装顺序)
- **并且** 记录关于冲突的警告
