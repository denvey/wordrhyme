# 核心启动规范 (Core Bootstrap Specification)

## 新增要求

### 要求：确定性启动流程 (Deterministic Bootstrap Flow)

系统应按照 `CORE_BOOTSTRAP_FLOW.md` 中定义的 7 个阶段执行确定的启动序列。顺序必须为：(1) 系统配置与内核，(2) 上下文提供者，(3) 插件清单扫描，(4) 插件依赖图，(5) 能力初始化，(6) 插件模块注册，(7) HTTP 服务器启动。

#### 场景：成功冷启动
- **当** 服务器在未安装任何插件的情况下首次启动时
- **那么** 所有 7 个启动阶段均按顺序执行
- **并且** 内核状态转换为 `running`
- **并且** HTTP 服务器在配置的端口上监听

#### 场景：成功热启动（带插件）
- **当** 服务器重启且安装了 2 个有效插件时
- **那么** 两个插件在第 3 阶段均被扫描
- **并且** 两个插件在第 6 阶段均被加载
- **并且** 调用两个插件的生命周期钩子 (`onEnable`)
- **并且** 系统达到 `running` 状态

#### 场景：无效插件被拒绝
- **当** 插件清单具有无效的 JSON 模式时
- **那么** 插件在第 3 阶段被标记为 `invalid`
- **并且** 插件未在第 6 阶段加载
- **并且** 系统继续进入 `running` 状态 (不会崩溃)

---

### 要求：内核状态管理 (Kernel State Management)

内核应维护一个状态机，状态包括：`booting`, `running`, `reloading`。状态转换必须记录日志。内核必须提供对当前状态的只读访问。

#### 场景：启动期间的状态转换
- **当** 服务器进程开始时
- **那么** 内核状态为 `booting`
- **当** 所有启动阶段成功完成时
- **那么** 内核状态转换为 `running`

#### 场景：重载期间的状态转换
- **当** 收到重载信号时
- **那么** 内核状态转换为 `reloading`
- **当** 重载完成时
- **那么** 内核状态恢复为 `running`

---

### 要求：执行阶段顺序强制执行 (Phase Ordering Enforcement)

启动阶段必须顺序执行。第 N+1 阶段在该第 N 阶段完成之前不得开始。插件代码在第 5 阶段 (能力初始化) 完成之前不得执行。

#### 场景：插件加载前上下文可用
- **当** 第 2 阶段 (上下文提供者) 完成工作时
- **那么** 租户、用户、语言区域、货币、时区上下文已注册
- **当** 第 6 阶段 (插件模块注册) 开始时
- **那么** 插件可以通过能力 API 访问上下文

#### 场景：阶段故障隔离
- **当** 第 1 阶段 (系统配置) 未能加载环境变量时
- **那么** 系统应记录错误并退出 (严重故障)
- **当** 第 3 阶段 (插件清单扫描) 遇到 1 个无效插件时
- **那么** 系统应将该插件标记为无效并继续
- **并且** 其他阶段正常进行

---

### 要求：启动日志记录 (Bootstrap Logging)

每个启动阶段必须记录其开始和完成。任何阶段中的错误必须记录上下文 (阶段名称、错误详情)。日志应为结构化格式 (JSON)，以便于观测。

#### 场景：成功的启动日志记录
- **当** 系统成功启动时
- **那么** 日志包含每个阶段开始的条目
- **并且** 日志包含每个阶段完成的条目
- **并且** 日志条目包含时间戳和阶段名称

#### 场景：错误日志记录
- **当** 第 3 阶段遇到无效插件时
- **那么** 创建一个结构化日志条目，包含：
  - `level: "error"`
  - `phase: "plugin-manifest-scanning"`
  - `pluginId: <id>`
  - `reason: <validation error>`
