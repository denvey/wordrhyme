# 集群协调规范 (Cluster Coordination Specification)

## 新增要求

### 要求：用于重载信号的 Redis Pub/Sub

系统应使用 Redis Pub/Sub 在集群节点之间广播重载信号。一个 `RELOAD_APP` 消息应在所有节点上触发优雅重载。

#### 场景：重载信号广播
- **当** 通过 API 安装插件时
- **那么** 服务器向 Redis 发布一条 `RELOAD_APP` 消息
- **并且** 订阅该频道的所有节点都收到该消息

---

### 要求：使用 PM2 进行优雅重载 (Graceful Reload with PM2)

每个服务器节点应监听 `RELOAD_APP` 消息。收到消息后，节点应触发 PM2 优雅重载。PM2 应执行滚动重启 (一次一个实例，零停机时间)。

#### 场景：触发滚动重启
- **当** 收到 `RELOAD_APP` 消息时
- **那么** 节点调用 `pm2 reload <app-name>` (如果 PM2 自动重启，则执行 `process.exit(0)`)
- **并且** PM2 在杀死旧实例之前启动新实例
- **并且** 在关机前完成活跃请求的排出

---

### 要求：共享插件存储 (Shared Plugin Storage)

`/plugins` 目录必须位于所有节点均可访问的共享存储 (NFS/NAS) 上。插件文件在所有节点上应保持一致。

#### 场景：插件在所有节点上可用
- **当** 在节点 1 上安装插件时
- **那么** 插件文件被写入共享的 `/plugins` 目录
- **当** 节点 2 重载时
- **那么** 节点 2 可以从共享目录读取插件文件
- **并且** 节点 2 成功加载插件

---

### 要求：开发模式支持

在开发模式下 (单节点，无 PM2)，重载机制可以禁用或简化。系统应检测 PM2 是否可用，并跳过集群协调。

#### 场景：开发模式跳过 Redis 广播
- **当** 服务器在没有 PM2 的情况下运行 (例如 `pnpm dev`) 时
- **那么** 插件安装不向 Redis 广播 `RELOAD_APP`
- **并且** 开发人员手动重启服务器
