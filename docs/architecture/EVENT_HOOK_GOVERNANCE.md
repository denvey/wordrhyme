# Nebula CMS — Event & Hook Governance (Frozen v1)

> 本文档定义 **Nebula CMS 事件（Event）与钩子（Hook）系统的最高治理规则**。
>
> 这是一个**治理文档（Governance Contract）**，不是 API 说明书。
>
> 它的目标不是“让插件更强”，而是：
>
> * 防止插件系统失控
> * 防止 Core 被事件反向绑架
> * 防止未来 1–3 年演进中出现不可收敛的耦合

---

## 0. 定位与原则（Non‑Negotiable）

### 0.1 事件系统的唯一目的

> **Event / Hook 只用于“扩展反应”，而不是“控制流程”。**

插件可以：

* 感知系统中发生了什么
* 在“被允许的时刻”附加行为

插件 **永远不能**：

* 决定 Core 是否继续执行
* 改变 Core 的主控制流
* 成为系统执行路径中的“必经点”

---

### 0.2 Core 永远是唯一的流程所有者

* Core 定义：

  * 事件是否存在
  * 事件在什么时候触发
  * 事件是否可中断

* Plugin：

  * 只能订阅
  * 只能响应
  * 永远处于 **被调用方（Passive）**

> ⚠️ Nebula **明确拒绝** WordPress 风格的“Hook 即流程”。

---

## 1. Event 与 Hook 的严格区分

### 1.1 Event（事件）

**定义：**

> Event 是 **事实广播**，表示某件事“已经发生”或“即将发生”。

特性：

* 单向广播
* 不可修改 payload
* 不可阻止主流程

示例（语义层面）：

* `content.created`
* `order.paid`
* `plugin.installed`

---

### 1.2 Hook（钩子）

**定义：**

> Hook 是 **受控扩展点**，Core 在明确位置允许插件“插入逻辑”。

特性：

* 明确声明
* 明确执行阶段
* 明确返回语义

Hook **不是**：

* 动态流程拼装工具
* 中间件系统
* AOP 框架

---

## 2. Hook 的治理模型（关键）

### 2.1 Hook 类型（冻结）

Nebula 只允许以下三类 Hook：

1. **Side‑Effect Hook**（默认）
2. **Transform Hook**（极少数）
3. **Decision Hook**（Core 专属）

任何新类型 Hook 都必须修改本文件。

---

### 2.2 Side‑Effect Hook（默认）

**规则：**

* 不返回值
* 不影响主流程
* 执行失败不会影响 Core

用途：

* 记录日志
* 发送通知
* 数据同步

这是 **99% 插件唯一允许使用的 Hook 类型**。

---

### 2.3 Transform Hook（严格受限）

**规则：**

* 只允许对“明确声明可变”的字段生效
* 返回新值，不能返回控制信号
* 执行顺序必须可预测

Transform Hook 必须：

* Core 明确声明
* 在文档中标注为 `@transform`

插件 **不能私自定义 Transform Hook**。

---

### 2.4 Decision Hook（禁止插件使用）

**定义：**

> 会影响 Core 是否继续执行的 Hook。

治理规则：

* ❌ 第三方插件禁止
* ❌ Marketplace 插件禁止
* ✅ 仅限 Core 内部模块

> 这是防止系统“被插件接管”的最后防线。

---

## 3. 执行顺序与隔离性

### 3.1 执行顺序规则

* 默认：

  * **无序执行**
  * 插件不得依赖顺序

* 如需顺序：

  * 由 Core 显式声明
  * 插件只能选择 `before / after` 阶段

---

### 3.2 插件隔离保证

* 单插件异常：

  * 不影响其他插件
  * 不影响 Core

* Core 必须：

  * 捕获所有插件异常
  * 降级而不是失败

---

## 4. 性能与稳定性治理

### 4.1 同步 / 异步策略

* 默认：

  * **异步执行**

* 同步 Hook：

  * 必须 Core 明确声明
  * 必须可超时

---

### 4.2 超时与熔断（强制）

* Core 为每个插件 Hook 设置：

  * 最大执行时间
  * 最大错误次数

* 超限后：

  * 自动降级
  * 标记插件为 `degraded`

---

## 5. 权限与安全约束

* Hook 回调：

  * 不自动拥有权限
  * 权限检查由 Core 完成

* Plugin 不得：

  * 在 Hook 中提升自身权限
  * 假设当前用户上下文存在

---

## 6. 版本与演进规则

* Event / Hook 名称：

  * 一旦发布，不得修改语义

* 新 Hook：

  * 只能新增
  * 不得复用旧名称

* 移除 Hook：

  * 必须经历 deprecate → remove

---

## 7. 明确禁止（Hard Ban）

Nebula **明确禁止**：

* 插件通过 Hook 改写 Core 状态机
* 插件之间通过 Hook 形成隐式依赖
* Hook 作为跨插件通信机制
* Hook 顺序作为业务逻辑假设

---

## 8. 设计哲学（写给未来的你）

> Hook 是“允许你做什么”，
> 不是“允许你接管什么”。

> 一个好系统的事件模型，
> 插件越多，Core 越稳定。

---
