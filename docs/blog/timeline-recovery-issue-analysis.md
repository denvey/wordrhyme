# Timeline 恢复策略问题分析

## 问题概述

文章《git reset --hard 后，我找回了 98.6% 的代码》中提到的 Timeline 恢复脚本存在一个严重的策略缺陷：

> **截止时间**设为 `2026-02-11 17:30`（事故发生时间），只取事故前的最新版本

这个策略对于**经常在 IDE 中打开的文件**是正确的，但对于**很久没有在 IDE 中打开的文件**，会导致恢复到非常旧的版本。

## 根本原因

**Timeline 只在文件被 IDE 打开并保存时才会创建快照。**

对于那些由 AI 助手生成但从未在 IDE 中打开过的文件，或者很久没有在 IDE 中打开的文件，Timeline 中的"事故前最后版本"可能是几周甚至几个月前的旧版本。

## 正确的恢复策略

### 策略 1：时间窗口检查（推荐）

```python
# 在恢复脚本中添加时间窗口检查
TIME_WINDOW_HOURS = 24  # 如果事故前最后版本距离事故时间超过 24 小时，给出警告

if (INCIDENT_TIMESTAMP - last_version_timestamp) / 1000 / 60 / 60 > TIME_WINDOW_HOURS:
    print(f"⚠️  警告: {file_path} 的事故前最后版本距离事故时间 {time_gap} 小时")
    print(f"   建议使用 Claude CLI 日志恢复，而不是 Timeline")
    # 跳过或标记为 WARN
```

### 策略 2：多源对比

对于时间窗口检查失败的文件：

1. **优先使用 Claude CLI 日志恢复**（如果有 Write 记录）
2. 如果 Claude CLI 日志也没有，考虑使用 Timeline 事故后的版本（如果它是从其他来源恢复后的正确状态）
3. 对比多个来源的版本，选择最合理的

### 策略 3：分级恢复

| 时间差 | 策略 |
|--------|------|
| < 1 小时 | 直接使用 Timeline 事故前版本 |
| 1-24 小时 | 使用 Timeline，但标记为 WARN |
| > 24 小时 | 优先使用 Claude CLI 日志，Timeline 作为备选 |
| > 7 天 | 必须使用 Claude CLI 日志或其他来源 |

## 修正后的恢复流程

```
1. 扫描 Timeline，找到所有文件的事故前最后版本
2. 对每个文件进行时间窗口检查：
   - 如果距离事故时间 < 24 小时 → 使用 Timeline 恢复
   - 如果距离事故时间 > 24 小时 → 标记为 WARN，优先使用 Claude CLI 日志
3. 对于 WARN 文件：
   - 检查 Claude CLI 日志是否有 Write 记录
   - 如果有，使用 Claude CLI 日志恢复
   - 如果没有，手动检查 Timeline 事故后版本是否正确
4. 生成恢复报告，列出所有 WARN 文件及其恢复来源
```
